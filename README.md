## 1 业务概述

### 1.1 系统定位

本系统用于**企业数据合规管理的评估问卷**场景：通过问卷的模板化设计、邮件分发、受访人填写与回收，对企业数据处理活动进行合规度量化评估，自动识别合规风险并纳入风险登记闭环，以满足 GDPR 等数据保护标准的可审计要求。

### 1.2 业务能力总览

业务层由三大功能域构成，均构建在通用后台基座（RBAC 权限、操作日志、数据权限、文件上传等）之上：

| 业务域 | 代码模块 | 核心职责 |
|--------|----------|----------|
| 问卷模板管理 | `business.questionnaire` | 问卷的可视化设计：题型、章节、题目、显示逻辑、风险规则、风险库与复用库 |
| 评估管理 | `business.assessment` | 问卷分发、受访人免登填写、答案回收、风险记录生成、审核闭环、过程监控 |
| 风险管理 | 风险记录 `as_risk_record`（`business.assessment`）+ 风险库 `qm_risk_lib`（`business.questionnaire`） | 评估风险记录的集中处理（查看/更新等级·处理状态·描述/删除）、风险库主数据维护与评分 |

### 1.3 模块关系

```
┌───────────────────────── 问卷模板管理 (qm_*) ─────────────────────────┐
│  QmTemplate ── QmChapter ── QmQuestion ── QmQuestionOption            │
│       │                         │            └── QmQuestionLogic(显示逻辑) │
│       │                         └── QmRiskRule ── QmRiskRuleCond(AND 条件) │
│       │                                  └── 引用 ── QmRiskLib(风险库)   │
│       ├── QmTemplateLibrary(行业模板库，JSON 快照)                       │
│       └── QmPersonalQuestion(个人题库，单题快照)                        │
└───────────────────────────────┬───────────────────────────────────────┘
                                 │ 模板发布后用于分发
                                 ▼
┌───────────────────────────── 评估管理 (as_*) ──────────────────────────┐
│  AsSurvey(分发实例) ── AsAnswer(答案,每题一行) ── AsAnswerHistory(变更史) │
│       │                  ├── AsComment(单题备注)                          │
│       │                  ├── AsRequest(单题修改请求)                      │
│       │                  └── AsRiskRecord(风险记录) ◀── 提交时由风险规则触发 │
└───────────────────────────────┬───────────────────────────────────────┘
                                 │ 风险记录由「风险管理」页集中处理
                                 ▼
┌──────────────────────────── 风险管理 ──────────────────────────────────┐
│  风险记录 as_risk_record：查看/更新等级·处理状态·描述/删除（不提供新增）    │
│  风险库 qm_risk_lib：风险主数据 + 固有/目标/剩余评分                       │
└────────────────────────────────────────────────────────────────────────┘
```

## 2 问卷模板管理

### 2.1 功能清单

| 子功能 | 说明 |
|--------|------|
| 模板创建 | 支持四种来源：空白 / 行业模板库 / 复制已有 / Excel 导入 |
| 模板状态管理 | 草稿（DRAFT）→ 已发布（PUBLISHED）→ 归档（ARCHIVED） |
| 问卷设计器 | 三模式可视化设计：问题配置、逻辑配置、规则配置；支持拖拽排序、跨章节移动 |
| 章节管理 | 模板—章节—题目三级结构，章节排序与显隐 |
| 题目管理 | 题型驱动渲染，复制 / 删除 / 收藏至个人题库 |
| 显示逻辑 | 依据某题答案，显示/隐藏目标题 |
| 风险规则 | 配置"满足条件即触发风险"，提交时自动求值 |
| 风险库维护 | 主数据：风险名称、等级、固有/目标/剩余评分 |
| 复用库 | 行业模板库（整卷快照）、个人题库（单题快照） |

### 2.2 模板与四种创建方式

模板（`qm_template`）记录名称、分类/行业、来源类型（`sourceType`）、状态（`status`）、外观样式配置（`styleConfig`，JSON）、说明。

| 创建方式 | sourceType | 实现要点 |
|----------|-----------|----------|
| 空白创建 | `BLANK` | 新建空模板，进入设计器从零配置 |
| 模板创建 | `TEMPLATE` | 从行业模板库选取，按 JSON 快照反序列化重建章节+题目 |
| 复制创建 | `COPY` | 深拷贝所选模板的章节+题目+选项，重新生成主键并重映射章节 ID |
| 导入创建 | `IMPORT` | EasyExcel 解析上传文件，按"章节/题型/题干/选项/必填"五列映射为题目 |

> 说明：复制创建**不复制显示逻辑**，因题目 ID 跨模板变更会导致逻辑条件失效；导入校验较宽松，题型与必填字段做容错归一化（兼容"单选题/必填/是"等写法）。

模板状态机：

```
DRAFT ──发布(须至少有题目)──▶ PUBLISHED ──归档──▶ ARCHIVED
```

### 2.3 问卷设计器（三模式）

设计器（前端 `views/questionnaire/designer/`）采用顶部切换、统一画布的结构：

| 模式 | 用途 |
|------|------|
| 问题配置 | 章节与题目的增删改、拖拽排序、题型新增（从左侧题型面板拖入章节） |
| 逻辑配置 | 配置题目间的显示逻辑（当条件题答案满足时，显示目标题） |
| 规则配置 | 配置风险规则（满足全部条件时触发指定风险库项） |

拖拽能力：原生 HTML5 Drag & Drop，支持题型→章节的新增投放、题目在章节间移动、章节内插入线精确定位；拖拽完成后统一重排全局 `orderNum` 并批量持久化。

### 2.4 题型体系

系统以枚举驱动题型渲染，当前实现 8 种题型（`QuestionConstants`）：

| 题型 | 常量 | 配置/存储方式 |
|------|------|---------------|
| 单选题 | `RADIO` | 选项子表 `qm_question_option`（optType=OPTION） |
| 多选题 | `CHECKBOX` | 选项子表 |
| 下拉题 | `SELECT` | 选项子表 |
| 单行文本 | `TEXT` | 题目列：最小/最大字数、正则、占位提示 |
| 日期 | `DATE` | 仅靠题型区分 |
| 是否题 | `BOOLEAN` | 仅靠题型区分（值 Y/N） |
| 文件上传 | `UPLOAD` | 题目列：最大文件数、单文件最大字节、允许类型 |
| 说明题 | `DESCRIPTION` | 纯展示，不参与校验与统计 |

> 设计约定：题型配置采用**结构化字段 + 子表**存储，明确不再使用 JSON（仅快照类实体模板库/个人题库仍用 JSON）。

### 2.5 显示逻辑

题目间逻辑（`qm_question_logic`）表达"条件→目标题→动作"：

- **语义**：当条件题（`condQuestionId`）的答案满足 `op condValue` 时，对目标题（`questionId`）执行 `action`。
- **操作符**：`EQ`（等于）、`NE`（不等于）、`IN`（属于，`condValue` 为逗号分隔多值）。
- **动作**：`SHOW`（显示）、`HIDE`（隐藏）。
- 一道目标题可配置多条逻辑；逻辑之间的组合（多条件全部满足/任一满足）交由前端解释。

> 本系统通过 SHOW/HIDE 控制题目可见性来间接实现"跳题"效果，未实现显式的分页跳转。

### 2.6 风险规则引擎（核心）

风险规则将"答案模式"映射为"风险库项"，是评估自动识别合规风险的核心机制。

**模型组成：**

| 实体 | 说明 |
|------|------|
| `qm_risk_rule` | 一条规则 = 模板 + 触发的风险库项 + 可选风险等级 |
| `qm_risk_rule_cond` | 规则条件子表，每条 = 条件题 + 操作符 + 条件值；**多条条件之间为 AND 关系** |
| `qm_risk_lib` | 风险库主数据，被规则引用（不复制内容） |

**规则配置（前端规则配置模式）：**
- 触发条件可由多行组成，每行选择"题目 + 操作符（等于/不等于/属于）+ 值"，行间标注"且"的关系。
- 触发动作选择风险库项；风险等级可选，缺省时回退到风险库自身等级。
- 选项题的条件值以选项下拉选择，文本题/IN 操作符以输入框填写。

**规则求值（评估提交时触发，见 §3.7）：**
1. 取本模板全部规则；
2. 将本次提交答案构建为 `题目ID → 答案值` 映射；
3. 对每条规则，遍历其全部条件，**全部条件命中（AND）才算规则命中**；
4. 单条件求值：`EQ` 等值、`NE` 不等、`IN` 答案属于条件值集合；其余返回不命中；
5. 命中即生成一条风险记录，等级取规则（缺省取风险库等级，再缺省取中）。

> 能力边界：求值为**纯字符串精确匹配**，不支持数值大小比较（如分数>阈值）、范围、包含等。

### 2.7 风险库与复用库

**风险库（`qm_risk_lib`）**：风险管理模块统一维护，字段含风险名称、类别、威胁、漏洞、处理计划、描述，以及固有/目标/剩余三档的影响、概率、评分。风险等级（HIGH/MID/LOW）按固有评分自动推导：

```
inherentScore ≥ 15  →  HIGH
inherentScore ≥ 8   →  MID
其余                →  LOW
```

**行业模板库（`qm_template_library`）**：整份问卷的 JSON 快照，"模板创建"方式从此还原。

**个人题库（`qm_personal_question`）**：单题快照（题+选项），按用户归属收藏，便于跨模板复用。

### 2.8 数据模型（问卷域）

| 表 | 说明 | 关键字段 |
|----|------|----------|
| `qm_template` | 问卷模板 | templateName、category、sourceType、status、styleConfig(JSON)、description |
| `qm_chapter` | 章节 | template_id、chapterName、orderNum、visible |
| `qm_question` | 题目 | template_id、chapter_id、questionType、title、required、orderNum、minLen/maxLen/regex/placeholder（文本）、maxCount/maxSize/accept（上传） |
| `qm_question_option` | 题目选项 | question_id、optType(OPTION/ROW/COL)、optLabel、optValue、orderNum |
| `qm_question_logic` | 显示逻辑 | question_id（目标）、condQuestion_id（条件）、op、condValue、action |
| `qm_risk_rule` | 风险规则 | template_id、ruleName、riskLib_id、level |
| `qm_risk_rule_cond` | 规则条件 | rule_id、condQuestion_id、op、condValue |
| `qm_risk_lib` | 风险库 | riskName、level、riskDesc、category、threat、vulnerability、固有/目标/剩余影响概率评分 |
| `qm_template_library` | 行业模板库 | 模板分类、模板快照(JSON) |
| `qm_personal_question` | 个人题库 | userId、题目快照(JSON) |

## 3 评估管理

### 3.1 功能清单

| 子功能 | 说明 |
|--------|------|
| 问卷分发 | 选择已发布模板与受访人（邮箱），生成填写令牌/链接，邮件发送 |
| 受访人填写 | 凭填写令牌免登填写（无需账号） |
| 草稿保存 | 填写过程中可保存草稿（不触发风险） |
| 答案提交 | 提交时校验、落库、生成风险记录 |
| 单题备注 | 题目维度给受访人的备注文案 |
| 单题修改请求 | 审核人/评估人就某题提出修改请求 |
| 退回重填 | 将答卷退回受访人重新填写，保留历史版本 |
| 审核闭环 | 审核通过 / 拒绝 |
| 过程监控 | 按模板聚合统计：分发数、回收率、完成率 |

### 3.2 评估实例状态机

评估实例（`as_survey`）以 `status` 字段为主线驱动流程：

```
              分发
 DISTRIBUTED ─────────▶ FILLING ────────────▶ REVIEWING ──────▶ APPROVED
   (已分发)    首次打开    (填写中)     提交      (审核中)  通过     (已通过)
                            ▲                       │ │
                            │ 退回填写 sendBack      │ └─拒绝──▶ REJECTED
                            └────────────────────────┤            (已拒绝)
                                                     │
                          returnBack 退回重填         ▼
                                              RETURNED
                                              (已退回) ──受访人再次提交──▶ REVIEWING
```

| 状态 | 常量 | 含义 |
|------|------|------|
| DISTRIBUTED | `STATUS_DISTRIBUTED` | 已分发，等待受访人打开 |
| FILLING | `STATUS_FILLING` | 受访人已打开填写 |
| REVIEWING | `STATUS_REVIEWING` | 已提交，待审核 |
| RETURNED | `STATUS_RETURNED` | 已退回，受访人重填 |
| APPROVED | `STATUS_APPROVED` | 审核通过 |
| REJECTED | `STATUS_REJECTED` | 审核拒绝 |

> 注：历史状态 `SUBMITTED` 已废弃，提交后统一进入 `REVIEWING`。

### 3.3 问卷分发

管理端选择已发布模板与受访人，发起分发：

1. 按受访人列表逐个生成评估实例（`as_survey`），初始状态 `DISTRIBUTED`；
2. 为每个实例生成填写令牌（`fillToken`，32 位 UUID）；
3. 记录受访者姓名/邮箱、评估人、截止日期、章节范围（可限定部分章节）、分发人、分发时间；
4. 异步发送 HTML 邮件，邮件正文含填写链接：`前端地址/#/app/survey/fill/{fillToken}`。

> 章节范围（`chapterScope`）为逗号分隔的章节 ID，空表示下发全部章节。

### 3.4 受访人免登填写

受访人无需账号，凭邮件中的填写链接进入填写页：

| 接口（白名单 `/app/**`） | 用途 |
|--------------------------|------|
| `GET /app/survey/fill/{token}` | 取卷：返回模板、题目（按章节范围过滤）、已填答案 |
| `POST /app/survey/submit` | 提交答卷（body 携带 fillToken） |
| `POST /app/survey/upload` | 上传附件（按 token 校验实例存在） |

首次取卷时，实例由 `DISTRIBUTED` 切换为 `FILLING`。

### 3.5 答案存储与变更历史

**答案表（`as_answer`）**：每题一行，原地更新。字段含答案值（JSON 文本）、单题备注、单题风险标记（`riskFlag`）。

**保存草稿与提交：**
- 保存草稿：仅 upsert 答案，记录变更历史，**不触发风险规则**；
- 提交：校验必填/正则/字数 → upsert 答案 → 生成风险记录 → 状态置 `REVIEWING`、写提交时间。

**变更历史（`as_answer_history`）**：当答案值或备注**实际发生变化**时记录一条 old→new；新增答案、清空答案、仅改风险标记等场景的留痕按变化判定。

### 3.6 单题备注与修改请求

| 功能 | 实体 | 说明 |
|------|------|------|
| 单题备注 | `as_comment` | 题目维度给受访人的备注，一题可多条 |
| 单题修改请求 | `as_request` | 审核人/评估人就某题提出的修改请求，前端以抽屉列表展示，按 `survey_id + question_id` 建索引 |

### 3.7 风险记录生成（双路径）

提交答卷时，风险记录（`as_risk_record`）有两条并行生成路径：

| 路径 | 触发方式 | 产物 |
|------|----------|------|
| 单题手动标记 | 答案 `riskFlag=Y` | 为该题生成一条风险记录（中等级、待处理） |
| 规则引擎自动 | §2.6 风险规则求值 | 每条命中规则各生成一条，等级取规则/风险库 |

风险记录字段：评估实例、题目、风险描述、等级（HIGH/MID/LOW）、处理状态（PENDING/PROCESSING/CLOSED）。

> 两套机制互不感知，一次提交可由二者共同产生多条风险记录。

### 3.8 审核闭环

| 操作 | 状态流转 | 说明 |
|------|----------|------|
| 退回重填（returnBack） | `REVIEWING → RETURNED` | 写退回原因至备注；受访人凭令牌重填后再提交回到 `REVIEWING` |
| 退回填写（sendBack） | `REVIEWING → FILLING` | 退回填写中状态 |
| 审核通过（review） | `REVIEWING → APPROVED` | 完成审核 |
| 审核拒绝（review） | `REVIEWING → REJECTED` | 完成审核 |

### 3.9 过程监控统计

按模板聚合该模板下所有评估实例，输出统计指标（`StatisticsVo`）：

| 指标 | 口径 |
|------|------|
| 已分发数（total） | 该模板评估实例总数 |
| 审核中数（submitted） | 状态为 REVIEWING 的数量 |
| 已退回数（returned） | 状态为 RETURNED 的数量 |
| 回收率（recoveryRate） | (审核中 + 已退回) / 已分发 |
| 完成率（completionRate） | 审核中 / (审核中 + 已退回) |

### 3.10 数据模型（评估域）

| 表 | 说明 | 关键字段 |
|----|------|----------|
| `as_survey` | 评估/分发实例 | template_id、respondentName/Email、fillToken、status、assessorIds、dueDate、chapterScope、distributeTime/submitTime、operUserId |
| `as_answer` | 答卷答案 | survey_id、question_id、answerValue(JSON)、remark、riskFlag |
| `as_answer_history` | 答案变更历史 | survey_id、question_id、oldValue、newValue |
| `as_comment` | 单题备注 | survey_id、question_id、content |
| `as_request` | 单题修改请求 | survey_id、question_id、content（索引 survey_id,question_id） |
| `as_risk_record` | 风险记录 | survey_id、question_id、riskDesc、level、handleStatus |

## 4 风险管理

风险管理模块对"评估产生的合规风险"做集中处理，并维护风险主数据。数据源为评估域的 `as_risk_record`（不再使用独立的 `risk_register` 台账）。

### 4.1 风险记录（as_risk_record）

风险记录由评估流程产生，有两种生成路径：

| 生成路径 | 触发时机 | 说明 |
|----------|----------|------|
| 风险规则引擎自动 | 评估提交时 | 命中模板风险规则（条件 AND 全部满足）即生成，等级取规则/风险库 |
| 评估详情页手动标记 | 审核人/评估人操作 | 对某题标记风险，填写等级与描述 |

「风险管理 → 风险记录」页对这些记录做集中管理：

- **查询**：按风险等级（HIGH/MID/LOW）、处理状态（PENDING/PROCESSING/CLOSED）、风险描述模糊检索；
- **展示**：风险描述、等级、处理状态、关联评估（标题+受访者）、关联题目（题干）、创建时间；
- **修改**：仅允许变更 等级 / 处理状态 / 风险描述（关联的评估与题目锁定不可改）；
- **删除**：单条或批量；
- **不提供新增**：风险记录只由评估流程产生。

> 接口路径 `/risk/register/*`、权限标识 `risk:register:*` 沿用原风险登记菜单（角色授权无需迁移），底层由 `AsRiskRecordController` 实现，列表分页后会关联补齐评估标题/受访者与题目题干。

### 4.2 风险库（qm_risk_lib）

风险主数据，被问卷的风险规则引用。字段含风险名称、类别、威胁、漏洞、处理计划、描述，以及固有/目标/剩余三档的影响、概率、评分。风险等级（HIGH/MID/LOW）按固有评分自动推导：

```
inherentScore ≥ 15  →  HIGH
inherentScore ≥ 8   →  MID
其余                →  LOW
```

> 评分模型（影响×概率，固有/目标/剩余三档）现统一由风险库 `qm_risk_lib` 承担；原 `risk_register` 的独立评分台账已移除。

### 4.3 数据模型（风险域）

| 表 | 说明 | 关键字段 |
|----|------|----------|
| `as_risk_record` | 风险记录（评估产生，风险管理页处理） | survey_id、question_id、riskDesc、level(HIGH/MID/LOW)、handleStatus(PENDING/PROCESSING/CLOSED) |

> 风险库 `qm_risk_lib` 字段详见 §2.8 问卷域数据模型。

## 6 关键业务规则与约定

1. **模板须发布方可分发**：草稿/归档模板不进入评估流程；发布前校验至少含一道题目。
2. **风险仅在提交时触发**：保存草稿不触发风险规则，避免未完成答卷产生噪声风险。
3. **风险规则条件为 AND**：一条规则的多个条件全部满足才命中。
4. **答案每题一行、原地更新**：变更走历史表留痕，不保留多版本答案行。
5. **填写令牌为唯一免登凭据**：分发后由 `fillToken` 标识一次填写，链接即凭据。
6. **复制模板不复制逻辑**：题目 ID 跨模板变更，逻辑条件会失效，故复制时丢弃显示逻辑。
7. **风险评分统一公式**：影响（1-5）× 概率（1-5），风险库额外按固有评分推导等级。
8. **风险记录双路径并存**：单题手动标记与规则引擎自动生成互不感知、可叠加。
9. **子表先删后插**：选项/逻辑/规则条件采用全量替换语义，前端传全量、后端幂等。
10. **审计字段统一**：所有业务表继承基类，含 `create_by/create_time/update_by/update_time/remark`。
